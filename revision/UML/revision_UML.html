Intro COO & UML
Pourquoi la conception objet ? Qu\'est-ce qu\'UML ?

L\'approche objet repose sur l\'idée que le monde réel est composé d\'objets qui interagissent.

Avant — approche fonctionnelle

1 application = ensemble de fonctions. Le système = flux de traitements. Couplage fort, réutilisabilité faible.

Avec la COO

1 système = objets qui communiquent. Mécanismes : encapsulation, héritage, polymorphisme, abstraction.

Avantages clés

  • Réutilisabilité — bibliothèques de classes, design patterns
  • Évolutivité — modifications localisées sans tout casser
  • Modélisation proche du réel — objets logiciels ≈ objets du domaine
  • Méthodologies — UP, Agile, XP supportent la COO

Unified Modeling Language = langage (pas une méthode !) fondé sur les notations graphiques.

  • Composé d\'un ensemble de notations graphiques pour représenter un système
  • Indépendant des langages de programmation
  • Standard OMG (Booch, Jacobson, Rumsbaugh — 1997)
  • Langage universel entre programmeurs, analystes, clients

Esquisse

Diagrammes informels à la main. Communication entre équipes.

Plan

Relation de contenance. Losange vide côté conteneur. Le composant peut exister sans le conteneur.

- - -▶ Dépendance

Multiplicités — Format

Combien d\'instances de B connaît une instance de A ?

NotationSignification
1Exactement 1
0..10 ou 1 (optionnel)
*0 ou plusieurs
1..*Au moins 1
n..mEntre n et m
2,5Exactement 2 ou 5

⚠️ La multiplicité se lit du côté opposé : la mult. côté B = combien de B une instance de A connaît.

Navigabilité

Dans quel sens peut-on "naviguer" d\'une classe à l\'autre ?

Bidirectionnelle (défaut)
Personne ←────→ Voiture

Personne a $mesTutures[] ET Voiture a $proprietaire.

Unidirectionnelle
Personne ─────▶ Voiture

Personne a $mesTutures[]. Voiture ne connaît PAS Personne.

Traduction en PHP

Pour Personne 1 ──possède──▶ 0..* Voiture :

// Voiture ne stocke pas Personne

Héritage & Classes Abstraites
Généralisation / Spécialisation — Polymorphisme — Override

Illustration — Héritage UML

Relation de Généralisation / Spécialisation :

  • Classe mère = super-classe = classe générale
  • Classe fille = sous-classe = classe spécialisée
  • Test : "B est une sorte de A" → héritage
  • En PHP : class B extends A

Contraintes possibles

{disjoint} — instances mutuellement exclusives

{overlapping} — une instance peut être dans plusieurs sous-classes

Animal
- nom : String
+ faireDuBruit() : String
△
Lion
+ faireDuBruit() {override}
Perroquet
+ faireDuBruit() {override}

Classe abstraite = ne peut pas être instanciée. Sert de modèle.

Notation : nom en italique ou <<abstract>>


Méthode abstraite = déclarée sans corps. Les classes filles doivent l\'implémenter.

Notation : méthode en italique

// PHP abstract class MoyenTransport { abstract public function seDeplacer(): string; } class Avion extends MoyenTransport { public function seDeplacer(): string { return "air"; } }

Étapes de modélisation

  1. Tester "est une sorte de"
  2. Factoriser / spécialiser les caractéristiques
  3. Vérifier l\'encapsulation des données
  4. Comportements différents par classe → polymorphisme + override
  5. Non implémentable dans la classe mère → abstract

Éléments clés

  • Instance : rectangle en haut (nom:Classe, souligné)
  • Ligne de vie : trait vertical pointillé
  • Activation : rectangle fin = objet actif
  • Création : flèche vers le rectangle de l\'instance
  • Destruction : croix ✕ en bas de la ligne de vie

Notation instance

// Nom seul paul : Acteur // Instance anonyme : ClasseA // Nouvelle instance créée obj2 := create() // Destruction destroy ✕

Message synchrone

L\'émetteur attend la réponse. Flèche pleine ─────▶. Le type le plus courant.

Message asynchrone

L\'émetteur n\'attend pas. Flèche ouverte ──────▷.

Message de retour

Réponse à un appel. Flèche pointillée - - - -▶. Peut être omis si implicite.

Message réflexif

Un objet s\'envoie un message à lui-même. Boucle sur sa propre ligne de vie.

Illustration — Fragments combinés : loop, alt, opt

sd

Nom du diagramme entier. Encadre tout.

ref

Référence à un autre DSeq (nommé).

loop

Répétition. Garde = condition / collection à itérer.

alt

Choix mutuellement exclusifs. Sections séparées par - - -. Équivalent if/else.

opt

Fragment optionnel. Équivalent if sans else.

par

Comportements parallèles.

critical

Région critique — pas d\'interruption possible.

break

Interrompt le fragment parent si garde vraie.

Exemple — loop

Pour chaque commande d\'une personne, calculer le total :

sd CalculerTotal Dupont:Personne cmd:Commande ────────────── ──────────── │ │ ┌─ loop [Pour chq cmd de mesCommandes] ─┐ │ │──────────────────────▶│ │ │ │ som += Total() │ │ │ │◀- - - - - - - - - - - │ │ └────────────────────────────────────────┘

Exemple — alt

┌─ alt [qualif = "Étudiant"] ────────────┐ │ som += forfaitEtudiant() │ ├────────────────────────────────────────┤ │ [else] som += forfaitNormal() │ └────────────────────────────────────────┘

Exemple — opt

┌─ opt [qualif = "Étudiant"] ────────────┐ │ som -= remiseEtudiant() │ └────────────────────────────────────────┘ // Si la condition est fausse : rien ne se passe

par & critical

par = comportements parallèles. Les sections s\'exécutent en même temps, ordre indifférent.

critical = région critique à l\'intérieur d\'un par. L\'opérateur l\'achemine en priorité.

Ingénierie directe
Situation : Une bibliothèque gère des livres et des adhérents. Un livre a un titre, un ISBN, un nombre de pages et une date de parution. Un adhérent a un nom, un prénom et une date d\'inscription. Un adhérent peut emprunter plusieurs livres, mais un livre n\'est emprunté que par un seul adhérent à la fois. L\'emprunt a une date de début et une date de retour prévue.
